# Recognizing string problems

> Good news: you already know half of this section. A string is an **array of
> characters** — so two pointers, sliding window, and the hash map (which you
> mastered in the Array section) show up again here. This section adds what is
> genuinely new about strings.
>
> As in the Array section, every pattern lesson opens with a **🎯 Try first**:
> solve it before seeing the theory. Feeling the difficulty is what makes the
> idea click.

## A string is an array of characters

`"abc"` is, in practice, `['a', 'b', 'c']` with indices `0, 1, 2`. Accessing
`s[i]` is O(1), scanning the string is O(n), and the array patterns work
almost identically. That's why String comes **right after** Array in the
course: it's the perfect place to consolidate those patterns in a new context
before moving on to more complex structures.

There's just one technical difference worth noting now: in many languages
(including Python and JavaScript) **strings are immutable**. You don't swap a
character in place — you create a new string. Building a string by
concatenating inside a loop can hide an O(n²) cost; when you need to build a
large output, accumulate it in a list and join at the end.

## What carries over from Array

| Array pattern | How it shows up in String |
| --- | --- |
| Hash map ("seen it?") | **character counting** — the frequency signature |
| Sliding window | **substring** that grows/shrinks, with a count map as its state |
| Two pointers | scanning the string from **both ends** (palindromes) |

## What's new in this section

| New tool | What for |
| --- | --- |
| **Frequency map** | a string's identity ignoring order (anagrams) |
| **Expand from the center** | find palindromes by growing from each center |
| **Stack** | match openings/closings, resolve the "most nested" first |
| **Encoding** | serialize data into text without ambiguity |

## The method stays the same

The Array section's playbook applies exactly here: **solve it by hand → brute
force → spot the waste → check the limits**. Only the repertoire of patterns
you consider in the "waste" step changes.

One string-specific detail is worth gold in the "brute force" step: generating
**all substrings** of a string costs O(n²) (there are n² of them), and
evaluating each one can add another factor. Much of this section is precisely
about not falling into that naive O(n²)/O(n³).

## The section's 10 problems

We'll use these Top 75 problems as **examples** of the patterns:

- Valid Anagram and Group Anagrams → *character counting*
- Longest Substring Without Repeating, Longest Repeating Character
  Replacement, Minimum Window Substring → *sliding window with frequency*
- Valid Palindrome, Longest Palindromic Substring, Palindromic Substrings →
  *two pointers and expand from the center*
- Valid Parentheses → *stack*
- Encode and Decode Strings → *encoding / design*

## Summary

- A string is an array of characters: indices, O(1) access, and the array
  patterns apply here.
- New in this section: frequency counting, expand from the center, stack,
  encoding.
- Watch out for the O(n²) of "all substrings" and the hidden O(n²) of
  concatenation in a loop.
