# What Is Big-O Notation

> This is the tool you'll use in **every** lesson of the course. Whenever we
> say "this solution is O(n)", we'll justify why — and after this section,
> you'll be able to justify it on your own.

## Why don't we measure in seconds?

Imagine you solved a problem and your code took 2 seconds. Is that fast or
slow? It depends: on your computer, on the language, on what else was
running... Seconds say nothing about the **algorithm** — they say something
about the machine.

That's why we measure something else: **how the number of steps grows as the
input grows**. That's the question Big-O answers:

> If I double the size of the input, what happens to the amount of work?

- Stays the same? **O(1)** — constant.
- Doubles along with it? **O(n)** — linear.
- Quadruples? **O(n²)** — quadratic.
- Grows just a little? **O(log n)** — logarithmic.

## Why this matters so much on LeetCode

Look at what happens to the number of steps when the array has 1 million
elements (a common size at LeetCode's limits):

| Complexity | Steps with n = 1,000,000 | In practice |
| --- | --- | --- |
| O(log n) | ~20 | instantaneous |
| O(n) | 1 million | instantaneous |
| O(n log n) | ~20 million | fast |
| O(n²) | **1 trillion** | ⏰ Time Limit Exceeded |
| O(2ⁿ) | a number with 300 thousand digits | impossible |

This is why so many problems have an "obvious" two-loop solution that
**works but doesn't pass**: it's O(n²). The right pattern brings it down to
O(n) — and learning these patterns is exactly the goal of this course.

## Big-O's three simplifications

Big-O is deliberately "lazy" — it ignores details that don't change the
shape of the growth:

1. **Constants drop.** O(2n) becomes O(n). Why? Because doubling the
   computer's speed also "multiplies by 2", and that doesn't change the
   algorithm. What matters is the *shape* of the curve, not its exact slope.
2. **Smaller terms drop.** O(n² + n) becomes O(n²). Why? With n = 1,000,000,
   the n² contributes 1 trillion and the n contributes 1 million — 0.0001% of
   the total. The dominant term swallows the rest.
3. **We think about the worst case.** If an algorithm can be fast when lucky
   but slow without that luck, Big-O describes the "unlucky" case. It's the
   guarantee you give: "never worse than this".

## The classes you'll run into

From fastest to slowest:

| Class | Name | Typical example |
| --- | --- | --- |
| O(1) | constant | accessing `nums[i]`, inserting into a hash map |
| O(log n) | logarithmic | binary search (halves the range each step) |
| O(n) | linear | scanning the array once |
| O(n log n) | linearithmic | sorting (efficient sort) |
| O(n²) | quadratic | comparing every pair (two nested loops) |
| O(2ⁿ) | exponential | trying every subset |
| O(n!) | factorial | trying every possible ordering |

Keep these three in mind especially, since they show up constantly across
the course's sections: **O(n)** is "I scanned everything once", **O(n²)** is
"I compared every pair", **O(log n)** is "I cut the problem in half each
step".

## Summary

- Big-O measures **how the work grows with the input**, not time in seconds.
- Constants and smaller terms drop; we think about the worst case.
- O(n²) with a large input = Time Limit Exceeded. The course's patterns exist
  to bring O(n²) down to O(n) or O(n log n).
